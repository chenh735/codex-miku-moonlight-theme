# Codex Miku Moonlight Theme GitHub 产品化设计

日期：2026-07-18  
状态：已完成对话确认，等待用户审核书面规格

## 1. 目标

将当前可运行的 Codex 初音未来月光都市主题整理为一个公开、可复用、可维护的 GitHub 项目。用户更换电脑或向其他人分享时，应能从 GitHub Release 下载 ZIP，解压并运行根目录安装脚本完成安装；开发者也能阅读源码、验证行为并继续维护。

项目名称和仓库名统一为 `codex-miku-moonlight-theme`。仓库创建在用户当前登录的 GitHub 个人账号下，公开可见，默认分支为 `main`，首个正式版本为 `v1.0.0`。

## 2. 已确认的产品边界

- 平台：Windows。
- 目标应用：官方 Codex 桌面版。
- 已验证 Codex 版本：`26.715.3651.0`；其他版本可能可用，但发布文档不得宣称已经验证。
- 运行依赖：Node.js 22 或更高版本。
- 默认视觉：初音未来月光都市主题，背景穿透透明度为 30%。
- 主题覆盖主页和对话任务页。
- 安装后通过“Codex 初音未来主题”快捷方式启动主题版 Codex。
- 不修改 Codex 安装包、`app.asar` 或用户的 `config.toml`。
- 不实现系统启动项、计划任务或长期驻留托盘程序。
- 代码公开，但主题中的角色图片及第三方素材不随 MIT 代码许可证自动获得再授权。

## 3. 方案选择

### 3.1 采用：完整独立主题仓库

仓库同时提供面向普通用户的一键安装入口、面向维护者的完整源码与测试、可审查的上游来源信息，以及自动生成的 Release 安装包。这一方案在易用性、透明度和长期维护之间最平衡。

### 3.2 未采用：保留上游仓库的产品结构

这种方式便于机械地同步上游，但现有上游名称、路径、托盘行为和预设说明与当前定制产品不一致，会使用户难以判断该运行哪个脚本，也容易继续传播过期文档。

### 3.3 未采用：只发布安装包

仅提供 ZIP 最简单，但用户无法方便地审查脚本，项目也缺少测试、来源和可维护结构，不适合公开共享。

## 4. 仓库结构

目标结构如下：

```text
codex-miku-moonlight-theme/
├─ README.md
├─ README.en.md
├─ Install.ps1
├─ Restore.ps1
├─ LICENSE
├─ NOTICE.md
├─ CHANGELOG.md
├─ UPSTREAM.md
├─ .github/
│  └─ workflows/
│     └─ release.yml
├─ windows/
│  ├─ assets/
│  ├─ docs/
│  ├─ scripts/
│  └─ tests/
└─ docs/
   ├─ installation.md
   ├─ screenshots/
   └─ superpowers/
      ├─ specs/
      └─ plans/
```

各组件职责：

- `README.md`：中文主文档，是仓库首页和普通用户的首要入口。
- `README.en.md`：与中文文档保持功能和风险说明一致的英文版本。
- `Install.ps1`：稳定的根目录入口，委托 `windows/scripts/` 中的实现执行安装。
- `Restore.ps1`：稳定的根目录恢复入口，只清理主题相关内容并恢复官方启动方式。
- `LICENSE`：软件源代码的 MIT 许可证。
- `NOTICE.md`：上游署名、商标、角色形象、第三方素材和非官方项目声明。
- `CHANGELOG.md`：按版本记录用户可见变化。
- `UPSTREAM.md`：保留上游仓库地址、导入范围和固定提交。
- `windows/assets/`：主题背景、图标等静态资源。
- `windows/scripts/`：安装、启动、恢复、验证和打包实现。
- `windows/tests/`：PowerShell 契约测试、产品测试及 Node.js 测试。
- `.github/workflows/release.yml`：标签驱动的测试、打包和 Release 发布流程。

根目录脚本是公共接口；内部脚本允许后续重构，但根目录安装和恢复用法应保持兼容。

## 5. 用户流程

### 5.1 普通用户安装

1. 打开 GitHub Releases。
2. 下载对应版本的 ZIP 和可选的 SHA-256 文件。
3. 解压 ZIP，右键运行根目录 `Install.ps1`。
4. 安装脚本检查环境并写入用户目录运行时。
5. 用户以后通过“Codex 初音未来主题”快捷方式启动。

### 5.2 设置与日常使用

- 默认透明度为 30%。
- README 和安装文档说明如何调整透明度，并说明修改后如何重新应用。
- 直接启动官方 Codex 不保证加载主题；文档必须醒目标明应使用主题快捷方式。

### 5.3 恢复官方状态

用户运行根目录 `Restore.ps1`。脚本只关闭主题相关进程、移除主题快捷方式和主题运行目录，不删除 Codex 的任务、项目、配置或其他用户数据。

## 6. README 与支持文档

中文 README 至少包含：

1. 项目简介和效果预览。
2. 功能列表与当前限制。
3. 系统要求和已验证版本。
4. Release ZIP 安装步骤。
5. 源码安装步骤。
6. 正确启动方式。
7. 透明度调整方法。
8. 更新、恢复和卸载方法。
9. 常见问题和故障排查。
10. 安全模型与本机 CDP 风险说明。
11. 上游来源、许可证和素材权利声明。

常见问题必须覆盖：

- 主页有主题但对话页视觉不明显。
- 通过默认 Codex 图标启动后主题未加载。
- Codex 升级后主题失效。
- Node.js 版本不足。
- 本机调试端口被占用或主题启动失败。

`docs/installation.md` 提供更详细的安装、设置、日志位置和恢复步骤。截图只展示当前实际产品，不用设计稿冒充已实现界面。

## 7. 发布自动化

发布由匹配 `v*` 的 Git 标签触发。首版标签为 `v1.0.0`。

工作流按以下顺序执行：

1. 检出标签对应的源码。
2. 配置 Node.js 22。
3. 运行现有 PowerShell 测试、产品契约测试和 Node.js 测试。
4. 验证安装包必需文件存在。
5. 使用文件白名单构建发布目录，禁止将日志、缓存、令牌、本机配置和开发工作树打包。
6. 生成 `codex-miku-moonlight-theme-<version>.zip`。
7. 计算并生成对应的 `.sha256` 文件。
8. 仅在前述步骤全部成功时创建 GitHub Release 并上传两个文件。

Release 内容至少包含中文安装摘要、支持环境、已验证 Codex 版本、升级提示和恢复入口。版本号采用语义化版本。

## 8. 错误处理与安全

安装脚本在以下情况显示明确的中文错误并停止，不留下半完成快捷方式：

- 找不到官方 Codex。
- Node.js 不存在或版本低于 22。
- 必需主题文件缺失。
- 本机端口启动失败。
- 运行目录不可写。

安全边界：

- 主题通过本机回环地址上的 CDP 控制 Codex 页面，因此拥有对当前 Codex 界面的较高操作能力。
- 脚本不应把调试端口暴露到外部网络。
- 项目不上传对话、任务或工作区数据到第三方服务器。
- 打包和测试必须检查仓库中不存在访问令牌、日志及本机专属配置。
- 恢复操作不得删除 Codex 用户数据。

这些声明用于清楚表达项目边界，不能消除公开传播第三方角色图片可能产生的版权风险。

## 10. 验证标准

整理完成后必须满足：

- 默认分支为 `main`，工作树干净。
- 根目录中英文 README 的安装、启动、透明度、恢复和风险说明一致。
- 文档不再把产品称为通用 `Codex Dream Skin`，也不再引导使用已移除的托盘或持久化行为。
- 已验证版本写为 `26.715.3651.0`，默认透明度写为 30%。
- 所有现有测试通过。
- 从白名单生成的 ZIP 通过内容检查。
- 从 ZIP 执行一次干净安装验证，主页和对话页均应用主题。
- `Restore.ps1` 能恢复官方状态且不删除 Codex 用户数据。
- Git 历史及发布包不包含令牌、日志、本机运行时设置或无关临时文件。
- 推送测试标签或正式 `v1.0.0` 标签时，GitHub Actions 能生成 ZIP、SHA-256 和 Release。
